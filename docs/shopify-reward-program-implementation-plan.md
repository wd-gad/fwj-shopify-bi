# Shopify Reward Program Implementation Plan

## 1. 今回の結論

現状の制約を踏まえると、直近は以下の 2 段階で進めるのが最も安全です。

- Phase 1: 月末一括でポイントを付与し、翌月から利用可能にする暫定運用
- Phase 2: 時間が取れる段階で、自動ポイント付与システムを実装する

今すぐ自動化を前提に進めるより、まずは「漏れなく、説明可能に、止めずに運用できる」状態を先に作るべきです。

## 2. 「Shopifyは小数点のポイント還元率を扱えない」は正しいか

結論として、これはそのままでは言い切れません。

- Shopify 公式ドキュメント上、`store credit` の付与額は金額として扱われ、Admin API では `MoneyInput.amount` が使われます
- Shopify の開発者向け型には `Decimal` があり、金額そのものは小数を扱える設計です
- そのため、「Shopify 全体の仕様として小数を扱えない」は一般論としては正確ではありません

ただし、次の可能性は高いです。

- 現在使っているポイント付与方法
- 担当者が見ている管理画面
- 既存のポイントアプリや還元率設定UI

これらが「整数 % のみ」を受け付ける仕様である可能性は十分あります。

つまり、確認結果は次の通りです。

- `Shopifyそのものが小数を扱えない` は誤解を含む
- `現在のポイント付与の設定方式では整数しか入れられない` はあり得る

今回の添付HTMLにあるような `5% / 7% / 8% / 12%` への丸め対応は、まさにこの「還元率UIが整数しか入らない」前提で作られているロジックでした。したがって、担当者の認識は「今の運用方式では実質的にそう」である可能性が高いです。

## 3. 直近の実務方針

時間がなく、購買システムは稼働中で、ポイント還元制度もすでにリリース済みである以上、当面は次の運用に寄せるのが妥当です。

- 毎月月末に対象会員へ一括でポイント付与
- 付与されたポイントは翌月 1 日以降に利用可能
- 付与ロジックは月次集計で確定させる
- 付与作業は手動または半自動で行う

この方式なら、次のメリットがあります。

- チェックアウト周りを今すぐ改修しなくてよい
- 本番の購買動線を止めない
- 計算式をあとから検証しやすい
- 返金や取消も月次確定前なら調整しやすい

## 4. 暫定運用でのポイント定義

月次付与では、ポイントを「注文時に即時確定」ではなく「月次で確定する還元残高」として扱います。

推奨定義:

- 対象注文: 当月中に完了した対象注文
- 対象商品: コンテストエントリー商品
- 還元率判定: 添付ロジックの `N = max(早期, 会員ランク, 競技数)`
- 現金値引き: 現行のストア実装どおり
- 月次ポイント: 注文ごとに `P = floor(Z * r / 100)` で算出し、月末に合算して付与
- 利用開始日: 翌月 1 日

この運用なら、小数の還元率そのものを checkout で扱う必要がありません。

## 5. 月末一括付与の実装イメージ

### 5-1. 最低限の運用

1. 当月注文を抽出
2. 対象注文だけを判定
3. 注文ごとの還元ポイントを計算
4. 会員ごとに当月ポイントを合算
5. 管理者が月末に一括付与
6. 翌月から利用開始

### 5-2. 付与手段

候補は 2 つです。

#### A. Shopify の store credit を使う

最も現実的です。

- 管理画面から手動付与できる
- API でも付与できる
- 後で半自動化しやすい

注意点:

- 顧客ログイン前提の運用に寄りやすい
- 「ポイント」ではなく「利用可能残高」に近い体験になる

#### B. 既存ポイントアプリ側で一括付与する

これも可能ですが、今回の「整数 % 制約」がこのアプリ由来である可能性があります。

注意点:

- 一括付与のAPIやCSV連携があるか要確認
- 還元率の制約は残る可能性がある

直近は `A. store credit` を第一候補に置くのが安全です。

## 6. 本番停止なしで実現できるか

可能です。停止は不要です。

理由:

- checkout に新しい自動付与処理を入れない
- 月末集計はバックオフィス作業として実施できる
- 付与前に集計結果を人が確認できる
- 問題があっても「今月分の付与を保留」にできる

本番影響は最小です。

## 7. 直近で必要な作業

### Phase 1. 運用設計の確定

先に以下を決めます。

1. ポイントの実体を `store credit` にするか
2. 対象商品はどこまで含めるか
3. 返品・キャンセル・返金時の扱い
4. 月末締め日時
5. 翌月利用開始の基準日時

### Phase 2. 月次集計ロジックの整備

このリポジトリの既存同期基盤を使って、月次集計用のスクリプトを作ります。

必要な処理:

- Shopify から当月注文を同期
- 対象注文を抽出
- 会員ランクを紐付け
- `S`, `N`, `X`, `Z`, `r`, `P` を注文ごとに計算
- 会員単位で当月ポイントを合算
- CSV または管理用一覧を出力

出力イメージ:

- 会員ID
- 顧客名
- メールアドレス
- 当月対象注文数
- 当月付与予定ポイント
- 付与根拠となる注文一覧
- 要確認フラグ

### Phase 3. 手動付与オペレーション

月末の標準手順を決めます。

1. 集計スクリプトを実行
2. 付与候補CSVを出力
3. 要確認案件を人が確認
4. 問題なければ一括付与
5. 実績CSVを保存
6. 翌月利用開始を案内

### Phase 4. 半自動化

時間が取れたら次を追加します。

- store credit 付与API連携
- 付与済みフラグ
- 二重付与防止
- 取消 / 再実行
- 監査ログ

### Phase 5. 完全自動化

その後に、注文確定時の自動計算と自動付与へ進みます。

## 8. 工数見積もり

前提:

- まずは月末一括付与
- このリポジトリの既存同期基盤を流用
- 本番 checkout は大きく触らない

### 暫定運用の立ち上げ

- 運用ルール確定: 0.5日
- 月次集計スクリプト作成: 1 - 2日
- CSV / レポート整形: 0.5 - 1日
- 手動付与手順書作成: 0.5日
- テスト集計: 0.5 - 1日

合計:

- 最短 3日程度
- 余裕込み 1週間以内

### その後の半自動化

- store credit API 連携: 1 - 2日
- 付与済み管理 / 冪等化: 1 - 2日
- 検証: 1日

合計:

- 3 - 5日程度

### 完全自動化

- 自動計算 / webhook / 例外処理 / 管理UI を含めて 2 - 3週間以上

## 9. 主要リスク

- いまの割引ロジックに撃ち漏らしがある
- 会員ランクの取得元が安定していない
- 早期申込条件の判定基準が商品側に載っていない
- 返金 / 取消分をどの月で相殺するか未定
- 既存ポイント運用と store credit の整合が未確認

## 10. 今回の推奨方針

優先順位は次の通りです。

1. まずは月末一括付与に切り替える
2. 付与媒体は `store credit` を第一候補にする
3. このリポジトリで月次集計スクリプトを作る
4. 集計結果をCSVで確認してから手動付与する
5. 落ち着いてから自動化に進む

この順なら、現在の本番運用を止めずに、短期間で現実的な運用へ持っていけます。

## 11. 今すぐ確認すべきこと

1. 現在の「ポイント」は実際には何で管理しているか
2. `store credit` を顧客向けのポイント残高として使ってよいか
3. 月末締めの対象日時をどうするか
4. 返品 / キャンセルは翌月付与前に除外するか
5. 既存割引ロジックの計算結果を月次集計で再計算してよいか
