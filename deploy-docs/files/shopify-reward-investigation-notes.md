# Shopify Reward Investigation Notes

## 調査日

- 2026-03-19

## 確認できたこと

### 1. ストアプラン

- ストアは `Shopify` プラン
- `Shopify Plus` ではない

### 2. 既存ポイント基盤

顧客メタフィールドに `loyalty` namespace が存在し、以下が確認できた。

- `loyalty.balance`
- `loyalty.tier_uid`
- `loyalty.easy_points_attributes`
- `loyalty.tier_maintenance_spend`
- `loyalty.tier_advancement_spend`

`easy_points_attributes` 内には以下が入っている。

- `percentage: 1.0`
- `point_value: 1`
- `currency_value: 100`
- `tier_name: "Default"`

したがって、現在のポイント運用は Shopify 標準のみではなく、Easy Points 系のロイヤルティ運用が既に入っている可能性が高い。

### 3. 商品タグ

コンテストエントリー系の商品には次のタグが見えた。

- `コンテストエントリー`
- `還元対象`
- `no-easy-points`

会員商品には次のタグが見えた。

- `FWJカード会員`
- `no-easy-discount`

つまり現行運用は、タグで「値引き対象」「Easy Points 除外」を振り分けている可能性が高い。

### 4. 注文の割引内訳

直近のコンテスト注文では、割引アプリケーションとして以下が確認できた。

- `1エントリー割引`
- `2エントリー割引`
- 割引コード `spoint3000`

例:

- 1エントリー注文
  - 自動値引き: `1エントリー割引`
  - 追加コード: `spoint3000`
- 2エントリー注文
  - 自動値引き: `2エントリー割引`
  - 追加コード: `spoint3000`

したがって、現在の値引きは単一ロジックではなく、少なくとも

- エントリー数に応じた自動値引き
- ポイント利用または別ロジック由来の `spoint3000`

が合算されている。

### 5. 3月分の集計結果

ローカル同期済みデータでの集計結果:

- 全注文: `236`
- コンテスト注文: `73`
- `還元対象` を含む注文: `68`
- `no-easy-points` を含む注文: `68`
- 割引済みコンテスト注文: `68`
- `spoint3000` 使用注文: `35`

補足:

- `還元対象` 注文と `no-easy-points` 注文が一致している
- つまり「ポイント対象の商品」でありながら「Easy Points の通常付与は除外」されている構造に見える
- これは、コンテスト商品にだけ別ロジックを当てるための運用である可能性が高い

## 現時点の仮説

1. 通常の Easy Points では `1%` の標準還元が存在する
2. コンテスト商品は `no-easy-points` で通常ポイント付与を止めている
3. その代わりに、別の割引ロジックで即時値引きを実現している
4. `spoint3000` はポイント利用または固定値引きコードとして別管理されている

## 実務上の示唆

- 「Shopify が小数を扱えない」というより、「現在の Easy Points ベース運用では整数％設定の制約がある」可能性が高い
- すでに本番運用中でタグ制御も複雑なため、今すぐ checkout まわりを全面改修するのは危険
- 当面は、コンテスト商品については月末一括付与へ寄せる方が安全

## 推奨アクション

1. Easy Points 管理画面で、還元率設定が整数のみかを確認
2. `1エントリー割引` `2エントリー割引` の作成元を確認
3. `spoint3000` が何由来かを確認
4. 月末一括付与では、`還元対象` かつ `no-easy-points` の注文を基準にする
5. 自動付与は後続フェーズで再設計する
